# AI 基础概念笔记

> Week 1 模块 A — 用自己的话整理的 AI 核心概念，建立后续学习 Agent、workflow、AI coding 和工具调用的共同语言。

---

## 1. LLM（大语言模型）

现在的大多数 LLM，抽象一点说就是每次猜下一个 token 应该生成什么，通过数学运算来算下一个出什么东西的概率更高。

**具体例子：** 闭源的 Claude、ChatGPT、Gemini，开源的 Qwen、Kimi 等等。你给它一段代码开头 `def fibonacci(n):`，它大概率能补完整个函数，因为训练数据里见过无数类似的代码模式。

**常见误区：** LLM 不是在"理解"你的代码，它是在做高维统计关联。所以它能写出看起来对的代码但逻辑上是错的——因为它不"懂"算法，只是"见过"很多算法。精确计算、实时数据、跨会话记忆，这些它都不擅长，别指望它当数据库用。

---

## 2. Prompt（提示词）

给模型的指令文本，决定了它以什么身份、用什么格式、做什么事。

**具体例子：** 同一个模型，你问"什么是 Gas？"它给你科普文章；但你加上"用 Solidity 开发者的视角，解释 Gas 优化策略"，输出就完全不一样。Prompt 就是给模型装"角色滤镜"。

**个人理解：** 早期有很重的 prompt engineering 文化，现在仍有人相信 prompt 调好很重要——当然很重要，但随着模型能力变强，提示词更多只剩方向性的作用。未来真正重要的应该是 harness，也就是围绕模型的限制与周边建设（上下文管理、工具编排、guardrails 等），这块会越来越重要。

---

## 3. Context Window（上下文窗口）

大模型能承载的上下文是有限的。简单来说，让它记太多东西会记爆。

**具体例子：** GPT-4o 大概 128K tokens，Claude 大概 200K tokens。如果你把一整本书塞进去，超出窗口的部分它就"看不见"了——不是忘了，是根本没读到。所以有 compact 的概念，快爆的时候压缩一下信息。

**使用边界：** 窗口大 ≠ 记忆力强。模型在长上下文中会出现"中间遗忘"现象（Lost in the Middle）——开头和结尾记得牢，中间容易被忽略。上下文窗口的扩展有点像大模型领域的摩尔定律，但似乎很快碰到瓶颈。在出现跨时代的上下文系统设计之前，用一些优化记忆和上下文结构的改进设计（RAG、compact、分层记忆）可能是主流出路。

---

## 4. Tool Use（工具调用）

LLM 本身只有输出文本的能力（当然也有多模态输出的模型）。本质上要让大模型真正参与工作，就得让它们学会用工具，来和外部世界交互。可以理解为大模型的 I/O 流水线——具体的功能由工具来承担。

**具体例子：** 模型本身不能查天气，但你给它一个 `get_weather(city)` 工具，它就会生成 `get_weather("Beijing")` 这样的调用请求，框架执行后把结果返回给模型，模型再基于结果继续推理。这就是从"说话"到"做事"的关键跳变。

**使用边界：** 工具调用的安全边界经常被低估。模型可能调错工具、传错参数、在不该调的时候调。比如你问"帮我看看这个合约安全吗"，它可能直接调了一个写入操作而不是只读操作。工具设计时要分清只读 vs 写入，写入类工具必须有确认机制。另外工具描述写得好不好直接影响模型选对工具的概率——这本质上是一种特殊的 prompt engineering。

---

## 5. AI Coding

从 Copilot 的代码补全，到 Cursor 刚开始的 tab 补全，到 Cursor 的 agent sidebar，再到 Claude Code——最核心的区别是人和 AI 在其中担当的位置在不断变化。

**演进过程：**
- **Copilot 时代：** 模型能力不足，只能做行级/块级补全，人写框架 AI 填细节
- **Cursor Tab：** 能理解当前文件上下文，补全更准，但还是人主导
- **Cursor Agent Sidebar：** AI 能全方位读项目里的代码，从全局视角来步入修改，但还是需要人来 review
- **Claude Code 时代：** AI 能力进一步增强，AI 主导、人为辅，甚至完全不看代码也能跑

**使用边界：** 不代表人可以完全放手。AI coding 工具在架构决策、安全审计、复杂业务逻辑上仍然不可靠。真正的价值不是"替代程序员"，而是把重复性编码的摩擦降到接近零，让人把精力花在更需要判断力的地方。

---

## 6. Agent（智能体）

让模型从"回答问题"进化到"自主规划、调用工具、多步执行"的系统。不只是对话，是做事。

**具体例子：** 你对 Hermes 说"帮我查一下 Web3-School 仓库的最新 commit，然后写个 summary 推到 Telegram"——它会自己调 GitHub CLI、读 git log、生成摘要、调 send_message。整个过程不是你一步步指挥的，是它自己规划的。

**使用边界：** Agent 的核心限制是它不知道自己不知道什么。它可能自信满满地执行一个错误操作。所以涉及资金、权限、不可逆操作的场景，必须有 human-in-the-loop 卡点。另外不是所有任务都需要 Agent——流程固定的用脚本就够了，Agent 的价值在于目标开放、路径不确定的场景。
